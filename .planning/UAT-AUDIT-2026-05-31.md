## Critical Gaps (Fix before production)

### CRIT-01: Draft entity leak via direct ID lookup
**File:** `src/app/api/entity/route.ts:62-78`
**Issue:** `GET /api/entity?id=ent-xxx` returns the entity without checking `publishedAt IS NOT NULL`. A draft or unpublished entity can be accessed by guessing its publicId.
**Evidence:**
```ts
const row = await db.query.entities.findFirst({
  where: eq(entities.publicId, id),  // NO publishedAt filter
});
```
**Fix:** Add `and(eq(entities.publicId, id), isNotNull(entities.publishedAt))` to the where clause.
**Risk:** Data breach — unpublished submissions exposed.

### CRIT-02: Rate limiting missing on 7 of 8 internal endpoints
**Requirement:** INTAKE-04 — "Rate limiting is distributed and covers all internal endpoints"
**Issue:** Only `POST /api/submit` has rate limiting. Missing on:
- `POST /api/review`
- `POST /api/reply/admin`
- `POST /api/upload`
- `POST /api/key-decisions`
- `GET /api/review`
- `GET /api/reply/admin`
- `GET /api/dashboard` (public but should still be rate-limited)
**Fix:** Add `rateLimitResponse()` to all internal API routes.
**Risk:** Brute force on reviewer endpoints, DoS on admin APIs.

### CRIT-03: Key decisions API has no authentication
**File:** `src/app/api/key-decisions/route.ts`
**Issue:** `POST /api/key-decisions` accepts decision creation/updates from any unauthenticated caller. No `getSession()`, no `hasRole()`, no `require2FA()`.
**Fix:** Add auth guards requiring `admin` role + 2FA on POST. Keep GET public (transparency).
**Risk:** Anyone can tamper with gated legal decisions.

## Medium Gaps

### MED-01: Submit API requires authentication — breaks anonymous submission
**File:** `src/app/api/submit/route.ts:43-46`
**Issue:** The API returns 401 if no session exists. But the form has an `isAnonymous` checkbox and the privacy policy claims anonymous submission is supported.
**Fix:** Remove the `requireAuthSession()` check or make auth optional. Store `actorId: null` for anonymous submissions.
**Risk:** Broken feature — users cannot submit anonymously as advertised.

### MED-02: Reply API matches unpublished entities
**File:** `src/app/api/reply/route.ts:20-24`
**Issue:** The fuzzy name match does not filter for published entities. A reply could be filed against a draft.
**Fix:** Add `isNotNull(entities.publishedAt)` to the entity lookup.

### MED-03: STATE.md is stale
**File:** `.planning/STATE.md`
**Issue:** Claims "0% v1.1", "Phase: Not started", "Status: Defining requirements". All 17 requirements are Done and deployed.
**Fix:** Update to reflect actual state.

## Low Gaps

### LOW-01: No input length limits on key-decisions POST
**File:** `src/app/api/key-decisions/route.ts`
**Issue:** `decisionId`, `category`, `title`, `description` have no max-length validation. Could be used for storage abuse.
**Fix:** Add Zod validation schema.

### LOW-02: Homepage shows stale static params for entity detail pages
**File:** `src/app/[locale]/page.tsx`
**Issue:** The `entity/[id]` pages still prerender `ent-001`, `ent-002`, `ent-003` from old demo data. If these IDs don't exist in the DB, the pages 404 at runtime.
**Fix:** Either remove static entity detail params or generate them dynamically from the DB at build time.

## Prioritized Human Test Plan

### Phase A — Security (Do first)

1. **Draft leak test**
   - Create a submission, do NOT publish it
   - Try `GET /api/entity?id=<draft-public-id>` without auth
   - Expected: 404. Actual: likely 200 with full data. **BLOCKER.**

2. **Rate limit bypass test**
   - Rapid-fire 20 requests to `POST /api/review` with valid reviewer cookies
   - Expected: 429 after threshold. Actual: all requests accepted. **BLOCKER.**

3. **Key decision tampering**
   - `curl -X POST /api/key-decisions -d '{"decisionId":"x","category":"legal","title":"x","description":"x"}'` with no cookies
   - Expected: 401. Actual: 200 with created decision. **BLOCKER.**

### Phase B — Feature Integrity

4. **Anonymous submission**
   - Open `/submit` in incognito, try to submit without logging in
   - Expected: Form allows submission. Actual: 401 error. **Fix MED-01 first.**

5. **End-to-end dual review flow**
   - Submit report -> Reviewer 1 approves -> Reviewer 2 approves -> Senior reviewer publishes
   - Verify entity appears on homepage and public record
   - Verify audit log entries exist in `review_logs`

6. **2FA enforcement**
   - Log in as reviewer WITHOUT enabling 2FA
   - Try to access `/reviewer`
   - Expected: Redirect to login or 403

7. **ClamAV file upload**
   - Upload a clean file -> should succeed
   - Upload EICAR test file -> should fail with MALWARE_DETECTED
   - If ClamAV not installed on server, verify graceful fallback (clean=true with warning log)

### Phase C — Edge Cases

8. **Right of reply against draft entity**
   - File reply using name of an unpublished entity
   - Expected: Reply stored but not linked, or 404. Actual: Linked to draft. **See MED-02.**

9. **Dashboard accuracy**
   - After publishing 1 entity and rejecting 1 submission, verify dashboard counts match

10. **Materialized view isolation**
    - Publish entity A, unpublish entity B
    - Query `published_entities` view directly in DB
    - Verify only A appears
