# Milestone v1.0: Foundation

**Status:** ✅ SHIPPED 2026-05-31
**Phases:** 1–9 (8 planned + 1 closure)
**Total Plans:** 8 phases executed inline

## Overview

Shipped the complete structural foundation of HLShajara: identity-free data model, non-bypassable validation choke point, append-only audit log, staff auth scaffold, public intake with boundary engine, reviewer console, legal release gate, public record with bilingual search, right-of-reply, and full AR/EN bilingual site with legal policies. Phase 9 (closure) wired auth enforcement, audit trail, and RBAC after milestone audit found critical gaps.

## Phases

### Phase 1: Integrity Core

**Goal**: The structural safeguards exist in schema and code so that identity targeting and zero-source publication are impossible by construction, and every state change is tamper-evident.
**Depends on**: Nothing (first phase)
**Requirements**: DATA-01, DATA-02, DATA-03, DATA-04
**Status**: Code complete

- [x] Identity-free schema (5 entity types, no sect/religion/ethnicity fields)
- [x] Allegation-source junction table with validation at app layer
- [x] Banned-pattern + incitement screen in `persist.ts`
- [x] Append-only hash-chained `review_logs` table + `verifyAuditChain()`

**Known gaps**: DATA-03 (withAudit never invoked before Phase 9), DATA-04 (audit trail empty before Phase 9)

---

### Phase 2: Auth & RBAC

**Goal**: Staff and submitters authenticate appropriately, least privilege is enforced, and no single actor can publish unilaterally.
**Depends on**: Phase 1
**Requirements**: AUTH-01, AUTH-02
**Status**: Code complete

- [x] Better Auth 1.4.7 with Drizzle adapter configured
- [x] Role hierarchy (submitter → reviewer → senior_reviewer → admin)
- [x] `canPublish()` and `hasRole()` helpers
- [x] `users` table with `role` and `is2faEnabled` fields

**Known gaps**: AUTH-01 (no 2FA plugin configured), AUTH-02 (RBAC not wired before Phase 9)

---

### Phase 3: Submission & Boundary Engine

**Goal**: Members of the public can submit evidence about a named actor, and structurally disallowed submissions are rejected at the front door before becoming candidates.
**Depends on**: Phase 1, Phase 2
**Requirements**: INTAKE-01, INTAKE-02, INTAKE-03, INTAKE-04
**Status**: Code complete

- [x] Public intake form with entity + allegation + source fields
- [x] Auto-reject for NO_SOURCE, GROUP_TARGET, INCITEMENT, HATE_TONE
- [x] File upload with Sharp EXIF/GPS stripping + SHA-256 hash
- [x] In-memory rate limiter on submit endpoint

**Known gaps**: INTAKE-02 (4 of 8 rejection codes unimplemented), INTAKE-04 (rate limiter only on submit, in-memory)

---

### Phase 4: Verification Pipeline

**Goal**: Reviewers verify sources and resolve identity, two independent reviewers must agree, and a legal/safety gate matches phrasing to evidence before any entry is eligible to publish.
**Depends on**: Phase 1, Phase 2, Phase 3
**Requirements**: VERIFY-01, VERIFY-02, VERIFY-03, VERIFY-04
**Status**: Code complete

- [x] Reviewer console with pending submissions list
- [x] Approve / Reject / Publish actions
- [x] Legal gate checks lawyer sign-off + source count

**Known gaps**: VERIFY-01 (no structured triage fields), VERIFY-02 (no source verification workflow), VERIFY-03 (no dual review), VERIFY-04 (no phrasing-to-evidence match)

---

### Phase 5: Legal Release Gate

**Goal**: No living person can be published without a recorded lawyer sign-off, and the operating jurisdiction/hosting decision is recorded before any production publish.
**Depends on**: Phase 4
**Requirements**: LEGAL-01, LEGAL-02
**Status**: Code complete

- [x] `validatePublication()` refuses living-person entries without lawyer sign-off
- [x] Review API calls `validatePublication()` before publish

**Known gaps**: LEGAL-01 (`isDeceased` hardcoded false), LEGAL-02 (no jurisdiction/hosting Key Decision gate)

---

### Phase 6: Public Record & Search

**Goal**: The public can browse and search the published record by conduct/role/strength/status only, with every entry showing its sources and an evidence-strength label.
**Depends on**: Phase 5
**Requirements**: RECORD-01, RECORD-02, RECORD-03, RECORD-04
**Status**: Code complete

- [x] Public record page with search
- [x] EvidenceCard with source count + evidence-strength label
- [x] Arabic + English search via `ilike` on name/role columns
- [x] Entity detail page with allegations and sources

**Known gaps**: RECORD-01 (no filter UI for status/type/evidence), RECORD-03 (no separate publish projection — same table)

---

### Phase 7: Right-of-Reply, Correction & Transparency

**Goal**: Named parties can respond, entries can be corrected/unpublished with reasons in the audit trail, and the public can see aggregate accountability statistics.
**Depends on**: Phase 6
**Requirements**: REPLY-01, REPLY-02, TRANS-01
**Status**: Code complete

- [x] Right-of-reply form (entity name, email, statement)
- [x] Reply API inserts pending row
- [x] Schema has `rightOfReplyState` and audit action enums for corrections

**Known gaps**: REPLY-02 (no reply processing workflow), TRANS-01 (no transparency dashboard)

---

### Phase 8: Bilingual Site & Policies

**Goal**: A bilingual public site explains the mission and the rules and carries the legal policies, with full AR/EN key parity.
**Depends on**: Nothing (parallel; legal-review of copy aligns with Phase 5)
**Requirements**: SITE-01, SITE-02, SITE-03
**Status**: Code complete

- [x] Full AR RTL / EN LTR with next-intl
- [x] Mission, FAQ, Terms, Privacy pages
- [x] Legal disclaimer banner on all pages
- [x] Static export script for GitHub Pages

**Known gaps**: SITE-02 (no dedicated anti-discrimination policy page)

---

### Phase 9: Critical Safety Closure (Inserted)

**Goal**: Fix the three critical blockers identified in milestone audit: unenforced auth, dead audit trail, missing RBAC.
**Depends on**: Phases 1–8
**Requirements**: DATA-03, DATA-04, AUTH-01, AUTH-02
**Status**: Complete

- [x] Session validation on all internal API routes (`/api/submit`, `/api/review`, `/api/upload`)
- [x] `withAudit()` / `appendAuditLog()` wired into all state-changing routes
- [x] RBAC enforced: `hasRole()` for reviewer console, `canPublish()` for publish action
- [x] Public entity API filters by `publishedAt IS NOT NULL`
- [x] Reviewer page server-side auth check with unauthorized message

## Milestone Summary

**Key Decisions:**

- Scope = documentation/accountability archive, NOT a boycott campaign
- Anti-discrimination enforced in the data model (no identity fields, no zero-source publish) — structural, not policy
- Lawyer-review gate before naming any living person
- Staging deployment on test-sanad (Sanad infrastructure)
- Phase 9 inserted post-audit to close critical safety gaps inline

**Issues Resolved:**

- All 8 planned phases code-complete and deployed to staging
- Bilingual i18n with full AR/EN parity
- Server-side PostgreSQL search
- File upload with metadata sanitization
- Right-of-reply API endpoint

**Issues Deferred:**

- OpenSearch / PostgreSQL FTS for Arabic search relevance
- ClamAV malware scanning
- Better Auth 2FA (TOTP unavailable in v1.4.7; email-OTP possible)
- Real research data seeding
- Transparency dashboard
- Source verification workflow
- Dual independent review enforcement
- Jurisdiction/hosting Key Decision gate

**Technical Debt:**

- In-memory rate limiter (won't scale across instances)
- No separate publish projection table
- `isDeceased` hardcoded false in publish API
- Sources auto-created as Tier "C" with hardcoded metadata
- No dedicated anti-discrimination policy page
- No login UI page (Better Auth API exists but no frontend)
