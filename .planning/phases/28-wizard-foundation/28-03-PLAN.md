<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser sessionStorage (shared-device exposure) | Draft holds submitter-entered values; on a shared device a later user could read `hls.submit.draft.v1`. |
| client state → eventual `/api/submit` | Client state is convenience; the server re-validates everything. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-05 | Info disclosure | `sessionStorage` draft on a shared device | mitigate | Use `sessionStorage` (cleared on tab close), NOT `localStorage`; `clearDraft()` on successful submit and on explicit "Start over" (wired in Plan 05). Surface the shared-device caveat in the restore-notice copy. No reviewer-only data ever enters the draft. |
| T-28-06 | Tampering | malformed/oversized JSON injected into the draft key | mitigate | `loadDraft` wraps `JSON.parse` in try/catch and returns null on failure (no throw); the reducer's `RESTORE_DRAFT` shallow-merges only known SubmitInput keys, never evals. |
| T-28-07 | Elevation/bypass | client reachability guard treated as a security control | accept | Reachability/branching is UX only; the server choke point (`validateSubmission`) is authoritative. Deep-link redirect-to-first-incomplete is correctness, not security. |
| T-28-SC | Tampering | npm installs | mitigate | No packages installed; pure TS + existing deps only. |
</threat_model>
