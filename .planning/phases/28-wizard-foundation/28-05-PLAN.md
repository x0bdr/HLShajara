<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `?step=` URL param (untrusted) | A user/attacker can set any `?step=` value, including unreachable or unknown ids. |
| sessionStorage draft (shared device) | Restored values are rendered back into inputs. |
| client interaction engine vs server | The wizard never publishes; everything routes to the existing `/api/submit` choke point (wired in Phase 31). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-10 | Tampering/EoP | crafted `?step=` to skip required steps | mitigate | On mount and on every `?step=` change, validate against `isReachable`; an unknown or unreachable id redirects to `firstIncompleteStep` (WIZ-06). Client guard is UX; the server choke point remains authoritative. |
| T-28-11 | XSS | restored draft values rendered into inputs | mitigate | Draft values bind to controlled inputs via React `value=` (auto-escaped); no `dangerouslySetInnerHTML`. `loadDraft` returns null on malformed JSON; `RESTORE_DRAFT` merges only known keys. |
| T-28-12 | Info disclosure | draft left on a shared device after submit | mitigate | `clearDraft()` on successful submit (skeleton kept here; real submit in Phase 31) and on explicit "Start over"; restore prompt copy notes the shared-device caveat. `sessionStorage` (not `localStorage`) bounds lifetime to the tab. |
| T-28-13 | Bypass | S1-S4 fields reintroduced via the scaffold | mitigate | The scaffold input renders ONE neutral placeholder text field — no phone/plate/address/map-pin/social/sect/identity/loyalty/profession input. Grep gate in `<verify>` asserts these tokens are absent from the shell+scaffold files. |
| T-28-SC | Tampering | npm installs | mitigate | No packages installed; React/next-intl/existing libs only. |
</threat_model>
