| client → /api/submit | The submitted SubmitInput (incl. leadNote) crosses to the authoritative server screen; the server remains the trust boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-07 | Tampering | rejection routing on a hostile/garbage `code` | mitigate | `resolveRejection` (Plan 01) returns `null` for any code outside the closed set; the null branch falls back to a generic error and never performs an attacker-chosen `goTo`. Resolved stepIds are real registry ids, so goTo always lands on a registered step. |
| T-31-08 | Spoofing/EoP | client-side gate (affirm + sources ≥ 2) | accept | Client gate is UX only; `validateSubmission` on `/api/submit` (WEAK_SOURCE choke point) remains authoritative — a bypassed client gate is re-screened and rejected server-side. |
| T-31-09 | Information disclosure | success reference id | accept | `submissionId` is an opaque sequential row id shown to the submitter who created it; no PII, no cross-record exposure. |
| T-31-10 | Information disclosure | draft on shared device | mitigate | `clearDraft()` (sessionStorage key hls.submit.draft.v1) fires on successful submit and on "Submit another", per S8; draft never persists past a completed submission. |
| T-31-SC | Tampering | npm/pip/cargo installs | mitigate | Zero new dependencies (constraint). No install tasks → no package-legitimacy gate needed. |
</threat_model>
