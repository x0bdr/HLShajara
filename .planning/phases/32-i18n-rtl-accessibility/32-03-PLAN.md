  <div className="choice-grid" role="radiogroup" aria-label={ariaLabel}>
    <div role="radio" aria-checked={checked} tabIndex={isActive ? 0 : -1}
         onKeyDown={handleKey} onPointerUp={...}>  // ArrowDown/Right→move(1), ArrowUp/Left→move(-1), Enter/Space→onConfirm
  Roving tabindex: only the active card is tabIndex 0; Arrow moves focus, Enter/Space confirms.
  The brass check span is aria-hidden="true" (decorative).

src/components/wizard/WizardPanel.tsx (focus + live-region pattern to verify on every step):
  <h2 ref={headingRef} tabIndex={-1}>{title}</h2>  + useEffect focuses it on [title,stepIndex,stepTotal] change.
  ONE persistent <div aria-live="polite" aria-atomic="true" {sr-only inline style}>{`${N} / ${M} — ${title}`}</div>.
  NOTE: there must be exactly ONE aria-live region in the whole wizard tree — if a step body
  (29–31) added its own aria-live, that is a duplicate-announcer defect this plan must resolve.

src/components/wizard/WizardProgress.tsx (aria-current pattern):
  active pill = <span className="wizard-step-pill on" aria-current="step">; done pills are <button>;
  upcoming are inert <span>. Progress conveyed by aria-current + text count, never color alone.

src/components/wizard/InputStep.tsx + the Phase-30 input steps (label/error a11y to verify):
  every <input>/<select>/<textarea> must have an associated <label for> or aria-labelledby;
  validation errors must be associated to their field via aria-describedby and carry icon+text
  (the .legal-error pattern), never color alone (UI-SPEC §11); the affirmation checkbox (Step 9)
  must be keyboard-reachable and labelled.

Reduced motion (verify, do not regress): the slide + the 200ms auto-advance delay are gated on
prefers-reduced-motion — CSS slide is disabled in hlshajara.css @media block; the auto-advance
delay is gated in WizardClient via matchMedia (Phase 28). Confirm both still hold.

Testing constraint: JSX cannot execute under --experimental-strip-types (Phase 28 precedent), so a11y
is verified via tsc + grep-gates over the markup + a clean `next build`, plus the staging screen-reader/
keyboard pass (CONTEXT — final sign-off is on staging, not local). NO new deps, NO test framework.
