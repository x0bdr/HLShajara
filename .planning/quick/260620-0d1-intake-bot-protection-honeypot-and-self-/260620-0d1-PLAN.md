<objective>
Harden HLShajara's public submission intake with two new bot defenses layered ON TOP of
the existing (verified-solid) reCAPTCHA v3 + Postgres rate limiter — which are NOT rebuilt:

1. A server-validated **honeypot** as the FIRST gate (hidden field; if filled → bot → no DB write, no leak).
2. A self-hosted **deterministic visible math challenge** with **gray-band escalation**: a low/missing/errored
   v3 score no longer hard-blocks — it returns `CHALLENGE_REQUIRED` with a fresh signed puzzle. A genuine human
   (Tor/VPN, low score) can ALWAYS complete by answering the puzzle. This is the one deliberate exception to
   fail-closed.

Purpose: Stop scripted spam without locking out anonymous/Tor submitters (the platform's mission requires
anonymous intake), and without adding any third-party data processor (PDPL).

Output: `src/lib/challenge.ts` (sign/verify), gray-band rewrite of `src/app/api/submit/route.ts`, honeypot
field through `WizardClient` + `submitSchema`, a bilingual RTL-safe `ChallengeGate` UI surfaced at ReviewStep,
full EN/AR i18n parity, `CHALLENGE_SIGNING_SECRET` env wiring, and a Vitest suite.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → POST /api/submit | Fully untrusted public intake; primary spam/abuse surface |
| challenge token (client-held) → verifyChallenge | Client may tamper/replay the signed token |
| client Content-Type / honeypot value | Bot-controlled; never trusted to be empty |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-0d1-01 | Tampering | challenge token | mitigate | HMAC-SHA256 over {answerHash,exp,nonce}; constant-time `crypto.timingSafeEqual` on verify; length-guard before compare. Tampered token → reject. |
| T-0d1-02 | Spoofing/Replay | challenge nonce | mitigate | Single-use: `checkRateLimit("challenge-nonce:<nonce>", {maxRequests:1})`; replay → allowed:false → reject. TTL-bounded (~5 min) so consumed nonces self-expire. |
| T-0d1-03 | Information Disclosure | honeypot existence | mitigate | Non-leaking response on honeypot hit (fake-success 200 OR generic 400; documented). No log/message that reveals the field, so bots cannot learn to skip it. |
| T-0d1-04 | Denial of Service | challenge issuance | mitigate | Rate-limit issuance via `checkRateLimit("challenge-issue:<ip>", {maxRequests:10/min})` on top of the existing IP submit limiter. |
| T-0d1-05 | Elevation/Bypass | gray-band accept path | mitigate | A verified challenge SUBSTITUTES for the v3 gate but still flows through submitSchema parse + advisory validateSubmission + withAudit — no validation is skipped. Single shared insert block. |
| T-0d1-06 | Information Disclosure | CHALLENGE_SIGNING_SECRET | mitigate | Read from env, fail-fast throw if unset (no insecure default); never logged; token payload contains only answerHash (not the answer) + exp + nonce. |
| T-0d1-07 | Tampering | npm install (vitest, vite-tsconfig-paths) | mitigate | Both are mainstream, widely-used dev-only tools (vitest, vite-tsconfig-paths) — verified-legitimate, dev-only, not shipped to prod runtime. No new prod/runtime captcha dependency added. |
| T-0d1-SC | Tampering | npm/pip/cargo installs | mitigate | Only two dev-deps added (vitest, vite-tsconfig-paths); both well-known. No [ASSUMED]/[SUS]/[SLOP] packages. |
</threat_model>
