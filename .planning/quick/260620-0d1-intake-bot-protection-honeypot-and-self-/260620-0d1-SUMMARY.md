## How the two defenses work

### 1. Honeypot (FIRST gate ÔÇö `src/app/api/submit/route.ts`)
- `submitSchema` gains an optional `website` field, deliberately **un-sanitized** and kept **outside
  the `.refine`** so the raw bot-supplied value's truthiness is what the route evaluates.
- The route destructures `website: honeypot` from the body **before** reCAPTCHA. A non-empty string Ôćĺ
  bot Ôćĺ **no DB write** Ôćĺ non-leaking response.
- Client renders a hidden `<input name="website">` that is **visually-hidden off-screen** (NOT
  `display:none`, which bots skip), `tabIndex=-1`, `autoComplete="off"`, `aria-hidden="true"`,
  positioned via logical `insetInlineStart` (RTL-safe). Real users never see/fill it.

### 2. Gray-band challenge (`src/lib/challenge.ts` + route + `ChallengeGate.tsx`)
- `generateChallenge()` returns `{ a, b, op, token }` ÔÇö structured operands (client renders the
  localized sentence) + a token = `base64url(payload).base64url(hmac)` where
  `payload = { answerHash: sha256(answer), exp: now+5min, nonce: uuid }`, HMAC-SHA256 signed.
- `verifyChallenge(token, answer)` order: constant-time HMAC (`timingSafeEqual`, length-guarded) Ôćĺ
  TTL Ôćĺ answer (Arabic-Indic normalized, hashed, compared) Ôćĺ single-use nonce via
  `checkRateLimit("challenge-nonce:<nonce>", { maxRequests: 1 })`. Any failure Ôćĺ `{ ok:false }`
  (never throws on bad input; the only throw is the fail-fast missing-secret guard).
- Route gray-band: a missing/low/errored v3 score no longer hard-400s. If the request carries a
  `challengeToken`+`challengeAnswer` and it verifies Ôćĺ it **substitutes for the v3 gate** and
  converges on the **same single insert block** (no validation skipped). Otherwise it rate-limits
  issuance per IP-hash (`challenge-issue:*`, 10/min Ôćĺ 429) and returns `CHALLENGE_REQUIRED` with a
  fresh puzzle. Wrong/expired answers re-issue a new puzzle (never a terminal block).

## Decisions (recorded per constraints)

- **Honeypot response shape:** fake-success-shaped HTTP 200 `{ ok:true, message }` with **no real
  row and no submissionId**. Chosen over a generic 400 because a bot cannot diff this against a
  legitimate success to discover that the `website` field is a trap ÔÇö the non-leaking property is
  stronger when the bot believes it succeeded. The response body never names "honeypot" or the
  field; verified by grep (no field name appears in any route JSON response).
- **CHALLENGE_REQUIRED transport:** HTTP **200** with a discriminated body
  `{ ok:false, code:"CHALLENGE_REQUIRED", challenge:{ a,b,op,token } }`. 200 (not 4xx/409) keeps it
  off the browser's network-error path and is the simplest signal for the client to branch on
  (`data.code === "CHALLENGE_REQUIRED"`); `ok:false` + `code` carry the real outcome.
- **Single-use nonce + issuance limiting:** both reuse the existing `rate_limits` table via
  `checkRateLimit` with distinct key namespaces (`challenge-nonce:<nonce>`, `challenge-issue:<ipHash>`).

## Deviations from Plan

1. **i18n script names:** the plan referenced `check:i18n` and `check:i18n:submit` ÔÇö both exist in
   `package.json` and were run as-is (no adaptation needed). No invented output.
2. **`@/db/schema` / conduct constants NOT mocked in route.test.ts (Rule 3 ÔÇö blocking issue):** the
   plan suggested `vi.mock("@/db")`. The route imports `@/lib/validation`, which uses `drizzle-zod`
   `createInsertSchema` over the REAL Drizzle table objects, and `schema.ts` builds `pgEnum` from the
   real `conductTypes`/`roleInConductTypes`. Fully mocking `@/db/schema` or `@/lib/constants/conduct`
   broke module load. Resolution: leave `@/db/schema` un-mocked (the route only reads `submissions`
   from it, harmless), and partial-mock `@/lib/constants/conduct` via `importOriginal` to keep the
   real constants while stubbing only `triageFromConduct`. This is a test-harness adaptation only;
   no production behavior changed.
3. **ChallengeGate CSS classes:** the plan said "use existing wizard classes." The wizard convention
   is `.form-field` + bare `<label>` + `.ds-input` + `.legal-error` (not the `ds-label`/`ds-error`
   names I first drafted). Aligned to the real classes; no new CSS added.
4. **`challengeAnswerLabel` i18n key:** added for parity and documented intent, but the visible
   `<label htmlFor>` (showing the question) is the accessible name, so the key is currently defined
   but not consumed in markup. Harmless; parity check passes (it validates key equality, not usage).
