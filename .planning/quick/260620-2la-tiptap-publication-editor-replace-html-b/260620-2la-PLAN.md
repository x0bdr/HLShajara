<objective>
Replace the raw "Body (HTML)" textarea in the publications authoring form with a TipTap
rich-text editor, and STRUCTURALLY eliminate the stored-XSS on the public publications page
by rendering only sanitized, schema-derived HTML.

Two structural guarantees:
1. AUTHORING — `posts.body` now stores TipTap doc JSON (stringified) entered through a
   bilingual, RTL-safe, accessible editor. The route rejects anything that is not a valid
   TipTap `{ type:"doc", content:[...] }` within a size bound (fail-closed; no more raw-anything),
   and walks every link mark href through `safeHttpUrl`, dropping unsafe hrefs on write.
2. PUBLIC RENDER — the public page no longer injects `post.body` raw. It runs body through a
   node-safe pipeline: TipTap JSON -> HTML via `@tiptap/static-renderer` (json/html-string,
   no DOM) -> `sanitize-html` strict allowlist. Legacy raw-HTML bodies go through the SAME
   `sanitize-html` allowlist. Either path, only allowlist-bound HTML reaches the DOM.

Purpose: a security-sensitive civic platform must not have a stored-XSS sink on its public
record. The fix is structural (schema-bound + allowlist), not a band-aid.

Output: TipTap deps installed; `publication-render.ts` + `publication-body.ts` libs (node-unit-tested);
hardened POST/PATCH route; `PublicationEditor` component; wired admin client + public page; EN+AR i18n.

VERIFIED PACKAGE (SSR JSON->HTML): `@tiptap/static-renderer@3.27.1` (MIT), import path
`@tiptap/static-renderer/json/html-string` — renders ProseMirror/TipTap JSON to an HTML string
with NO DOM / NO jsdom (pure node). Peer deps `@tiptap/core@3.27.1` + `@tiptap/pm@3.27.1` are
satisfied by the installed TipTap set. This is why no DOM shim or `@tiptap/html` is needed.
</objective>
