“New Report” Submission Taxonomy — Constraint & Safety Audit

Multi-model review of the proposed 8-sector report form, grounded against the platform’s codified hard constraints.
Date: 2026-06-14 Scope: proposed report taxonomy + full specs.JPG + CONCEPT.md + live i18n copy Method: Claude workflow (7 agents, repo-grounded) + Codex gpt-5.5 (high) + Kimi — independent, then synthesized Models concur: 3 / 3 → do not ship as designed

Verdict — Reject as designed; one finding overrides all others

The form is the intake front-end for a sect-based collective-punishment program. The project’s own concept layer (CONCEPT.md), README, the campaign poster (full specs.JPG), and the live bilingual site copy define the campaign as targeting the Alawite community as a sect — “the nurturing environment / البيئة الحاضنة” — and call to boycott their shops, taxis, schools, universities, hospitals, and even “humanitarian dealings.” The 8 report sectors map one-to-one onto that poster’s “Social Isolation Domains” grid.

This is collective punishment by identity. It directly contradicts the platform’s own charter (“no group, no identity-based targeting”) and the anti-discrimination logic its code already enforces (DATA-01 identity-free schema, GROUP_TARGET / INNOCENT_PARTY screens). It is ethically indefensible and legally fatal. No partial salvage of this framing is possible — it must be removed.

Below: (1) the overriding finding with evidence; (2) the form-level findings; (3) the multi-model consensus matrix; (4) the lawful redesign that fits your existing schema; (5) a live-contamination remediation checklist.

2 · Form-level findings (independent of the campaign frame)

Even if the sectarian frame were removed, the proposed taxonomy as drawn still breaks multiple codified constraints. Severity-ranked; each maps to a real enforcement point in your code.

Critical F1 · Mass-targets ordinary private individuals mission · INNOCENT_PARTY

Fields باعة جوالة (street vendors), delivery/taxi/private drivers, عاملة تنظيف / عاملات تنظيف (cleaners), اكشاك (kiosks), reception/front-desk, labour, employees, nurses, instructors, students.
Breaks Mission scope (“named individuals tied to specific conduct with credible public evidence”); INNOCENT_PARTY / INNOCENT_PROFESSIONS already shields doctors, nurses, teachers, students (persist.ts:83–100). The entity-type enum has no slot for a taxi driver or cashier.
Harm Turns the archive into a public blacklist of low-power workers — retaliation, employment blacklisting, stalking, score-settling. These are not presumptive perpetrators of Syria crimes.
Fix: Remove all ordinary-worker categories. Individuals are admissible only when a credible public source already ties a named person (typically an official/commander or a publicly-named perpetrator) to specific documented conduct.

Critical F2 · Political-loyalty screening & “bad attitude” complaint board non-sectarian · GROUP_TARGET · GDPR Art.9

Fields Tourism Sub-type B hint: “Does he support asad regime? Report for bad attitude.” · the شبيح person-category · المتعاونة والمتعاطفة (“collaborating/sympathising”).
Breaks Non-sectarian tone; “justice not revenge / documentation not harassment”; GROUP_TARGET; GDPR Art. 9 / PDPL special-category (political opinion of an identifiable person).
Harm The single most dangerous string in the draft: it instructs users to denounce a private person for perceived loyalty and for interpersonal rudeness. Pure grudge/loyalty fuel, with no conduct and no source.
Fix: Delete the hint and the “support/sympathy/collaboration” framing entirely. Loyalty and “attitude” are never reportable. Replace with a closed list of documented conduct types that each require a source.

Critical F3 · No-source intake — “تجربتك” as the evidentiary basis INTAKE-01 · validateSubmission

Fields Global Step 4 “تجربتك (your experience) + supporting-documents checkbox.”
Breaks “No source, no publication.” Your code already enforces the opposite of this design: NO_SOURCE if 0 sources and WEAK_SOURCE if < 2 (persist.ts:128–146); Zod sourceLinks.min(1) (validation.ts:73). A personal anecdote is explicitly not a source.
Harm Trains submitters to accuse from grievance; floods intake with uncheckable claims about named people; destroys credibility before courts / UN COI / IIIM, which require source citations.
Fix: Make a credible-source citation the primary required field (URL/doc/upload), minimum two independent third-party sources. Personal experience may be captured only as a non-public lead note, never sufficient to publish. Replace the checkbox with a validated source field.

Critical F4 · Doxxing PII — plates, phones, addresses, map pins PRIVATE_TARGETING

Fields street / nearest location / Google-Map pin · phone & driver phone · car plate · taxi number · LinkedIn/Indeed/social handles · real-estate houses/villas/land + owner name.
Breaks PRIVATE_TARGETING (persist.ts:91–104) already rejects phone/GPS/street-address/social-handle patterns; PROJECT.md “no home addresses / location-tracking.”
Code gap surfaced PRIVATE_DATA_PATTERNS does not yet match car plates or international phone formats (+\d{7,}, 00…) — a plate pasted into allegationDescription passes today. The proposed field would make doxxing a first-class column.
Fix: Remove all individual phone/plate/residence/map-pin/social fields. Allow only coarse location (country / governorate / city) for the site of conduct, never residence. Add plate + international-phone patterns to PRIVATE_DATA_PATTERNS with Vitest cases.

Critical F5 · Free-form “شبيح” label = accusation-by-epithet GROUP_TARGET · LEGAL-01

Field Individuals Sub-type A → “شبيح (shabih)” as a selectable person type.
Breaks “Named individual tied to specific conduct”; identity/collective epithet not a documented act; bypasses the per-claim source FK (a label is not an allegationSources row). Living-person naming also needs the LEGAL_GATE lawyer sign-off.
Fix: Remove the label/epithet field. Conduct is entered only as a sourced allegation; categories, if any, are a closed enum of acts reviewed by counsel. Add شبيح/عميل/مخبر to BANNED_PATTERNS so they’re caught inside free text too.

High F6 · The lawyer-gate is a self-asserted boolean, not a counsel artifact existing build · LEGAL-01

Where hasLawyerSignOff arrives in the publish payload; validatePublication() (persist.ts:237–246); publish action (api/review/route.ts:182–192).
Harm The platform’s entire defamation/Art. 9 defence reduces to body.hasLawyerSignOff === true, set by the same reviewer who publishes — no named lawyer, jurisdiction, scope, or signed artifact. In litigation, an unverifiable boolean is an admission the control was theatre.
Fix: Make sign-off a FK to a persisted lawyer_sign_off audit row (lawyer identity, bar/jurisdiction, entry IDs, date, standard version) created by a distinct legal role and hash-chained in reviewLogs (the lawyer_sign_off action enum already exists). Publish reads it from that row, never from request input.

High F7 · The ≥2-source gate checks count, not credibility or independence existing build · editorial

Where WEAK_SOURCE floor (persist.ts:139–146) counts links; two blog posts — or two copies of the submitter’s own anecdote — pass.
Harm A private person can be named and published behind any two arbitrary URLs. For private (non-public-figure) persons, defamation law gives the highest plaintiff protection — and the platform, not the anonymous submitter, is the publisher/defendant.
Fix: Encode source tier (A/B/C) into the publish gate, not just count. entityType='individual' of low authority must require Tier-A/B (court/UN/sanctions/recognized HR org). The submitter’s own narrative never counts toward the threshold. Add a public-figure/authority nexus test as a publish precondition.

High F8 · No lawful basis / Art. 9 condition / DPIA / retention; isAnonymous defaults false GDPR/PDPL · existing build

Harm The platform processes political-opinion data about identifiable living persons with no recorded Art. 9(2) condition, no DPIA, no retention limit — rejected submissions still store the named person + unverified anecdote indefinitely. isAnonymous defaulting false makes a submitter-identity DB (a coercion/subpoena target) the path of least resistance; ipHash is pseudonymisation, not anonymity.
Fix: Persist a versioned lawful-basis + Art. 9 condition record (the key_decisions table) referenced by FK from every published entity; commission a DPIA; add a retention policy that purges identifying free-text from rejected submissions; default isAnonymous to true; demote ipHash to a short-TTL rate-limit token off the permanent row; provide an access/erasure/right-of-reply path reachable without an account.

High F9 · Taxonomy: sector-axis is mission-drift; no conduct vocabulary; overlaps IA · data-model

Issues The 8 consumer sectors (Commercial/Tourism/Medical/Real-Estate…) read as a complaint board, not a crimes archive; the same actor (cleaners, delivery apps) recurs across categories; there is no conduct/“report-type” taxonomy — everything normalises to one free-text allegationDescription, leaving triageCategory unpopulated and allegationClassification an orphan field.
Fix: Replace sectors with an accountability taxonomy: one orthogonal actorClass (individual | entity, with optional entity subtype) + one required closed conductType enum (arbitrary_detention | torture | extrajudicial_killing | enforced_disappearance | sexual_violence | property_seizure | command_responsibility | …) rendered first and populating triageCategory. Conduct-based ⇒ fully DATA-01-compatible.

Medium F10 · Free-text location/status reconstruct targeting & overstate certainty existing build

Issues allegationLocation (free text) can hold “his house in [village]” — re-introducing the residence-targeting the GPS regex strips; evidenceLevel/status are reviewer self-attestation (a private person can be labelled indicted/convicted with no court-record FK — per-se defamation).
Fix: Constrain allegationLocation to governorate/city for individuals and screen it with the private-targeting regex; gate status/evidenceLevel ≥ Tier-A behind a required matching-tier sources FK.

Medium F11 · i18n — mixed-language labels, loaded terms, FAQ contamination is bilingual EN/AR parity

Issues Fields mix Arabic + English in one label (“name/nickname Phonenubmer position”); typos (“Phonenubmer”, “recaption”→reception, صيديلة, جميعات); colloquial/loaded terms (شبيح, كنية/فانوس, “bad attitude”) lower evidentiary seriousness; the sectarian FAQ a1 is contaminated in both en.json and ar.json.
Fix: One clean legal label per language with full EN/AR parity; a controlled bilingual glossary; remove slang and epithets; run check:i18n after the rewrite.

4 · What’s salvageable — the lawful redesign

The 5-step wizard shell and the named-entity documentation core are keepers. Rebuild the content from the schema up, source-first, conduct-based — which is exactly what submissions → entities → allegations → sources and validateSubmission already want.

  1. Actor identity — Who, and which class? individual | entity (+ entity subtype: organization / military_unit / security_branch / official_body). Individuals only when a public source already names them. No cleaners, drivers, students, nurses, vendors, kiosks, “ordinary employees.”
  2. Conduct — Required closed enum (torture, enforced_disappearance, extrajudicial_killing, arbitrary_detention, sexual_violence, property_seizure, command_responsibility, sanctions-listed conduct, …) + a description tied to that act. Never identity, loyalty, or “attitude.”
  3. Sources — two source rows by default, with live count and “minimum 2 independent sources” helper. Required source type (UN/IIIM · court · sanctions list · recognized HR org · corroborated journalism · official filing). Reject “my experience” / “social post” as a source. Submitter narrative never counts toward the threshold.
  4. Context, safe granularity — Governorate/city + date range only. No street, map pin, plate, phone, residence. Inline warning mirrors PRIVATE_DATA_PATTERNS so the submitter is corrected before submit, not rejected after.
  5. Submitter / anonymity — isAnonymous defaults true; checking it clears & disables name/email client-side and the server drops them. Contact optional, purpose-labelled (reviewer follow-up only), stored separately. Nothing publishes until dual review + counsel-artifact sign-off for living persons.

6 · What’s already strong (keep it)

For balance — the engineering is genuinely principled, which is exactly why the concept-layer drift is worth correcting rather than abandoning:

  • keep Identity-free schema (DATA-01) — no sect/ethnicity/family columns exist; violation is structurally impossible.
  • keep Source-required intake at three gates (Zod min(1) → validateSubmission ≥2 → publish sourceCount>0).
  • keep Dual independent review state machine + RBAC + mandatory 2FA for staff.
  • keep PRIVATE_TARGETING / INNOCENT_PARTY / GROUP_TARGET / INCITEMENT / HATE_TONE screens, EXIF/GPS stripping (sharp), hash-chained immutable audit log.
  • keep Living-person lawyer gate concept (just make it a real artifact, F6).

The lawful version of this platform — named individuals/units/officials, tied to documented conduct, backed by credible public sources, gated by dual review and counsel — is important, defensible accountability work. This report rejects the collective-punishment direction precisely to protect that core.
