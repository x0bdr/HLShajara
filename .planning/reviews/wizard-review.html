HLShajara Submit Wizard — Full Step Review

Scope: all 8 wizard steps across all category/subtype flows.
Goal: eliminate duplicate fields and irrelevant data in every step.

Executive Summary

The initial complaint — "review page shows driver name / taxi fields for a travel company" — is already fixed. The deeper review found two critical gaps and several duplication issues across the wizard:

  • Source links are never collected in the active wizard, even though the review and API expect them.
  • Several detail flags are clickable but do nothing (name, phone, doctor, instructor, student, etc.).
  • Duplicate data capture exists between entity-type-name and report-details for individuals and vehicle flows.

🚨 Critical Issues

Critical Source links (sourceLinks) are never collected

Files: MediaEvidenceStep.tsx, WizardClient.tsx, EvidenceStep.tsx, rejection-map.ts

The active media-evidence step only collects a single optional mediaLink, file uploads, and notes. The sourceLinks array that the review screen renders and the API validates is never populated. A dedicated EvidenceStep.tsx component exists but is not rendered.

What the intro promises:
introRule2: "At least two independent credible public sources"
What the review currently shows:
Sources: — (empty because sourceLinks.length === 0)
Suggested fix:
Render the source-link repeater from EvidenceStep.tsx inside MediaEvidenceStep.tsx (or replace the step). Populate form.sourceLinks with URL, optional title, and source type.

Critical Detail flags that do not map to any field

Files: category-config.ts, ReportDetailsStep.tsx

Several categories expose selectable flag cards, but DETAIL_FLAG_FIELDS has no mapping for them. Selecting the card updates state but never reveals an input and never affects review.

CategoryUnmapped flagsWhat users see
individualsname, phone, position, socialCards that look active but produce no fields
educationalinstructorCard selected, no input shown
medicaldoctor, nurseCards selected, no input shown
organizationsstudentCard selected, no input shown
Suggested fix:
Either add mappings in DETAIL_FLAG_FIELDS (e.g., name → reportedPersonName) or remove the unmapped flags from each category's detailFlags array.

⚠️ Medium Issues — Duplication & Gaps

Medium Individual name is collected twice

Files: EntityTypeNameStep.tsx, ReportDetailsStep.tsx, category-config.ts

For the individuals category, the reported person's name is asked in step 2 (entityName) and again in step 4 (reportedPersonName).

Step 2 label:
etnNameIndividual: "اسم الشخص المُبلَّغ عنه"
Step 4 label:
detailsReportedName: "الاسم / الكنية المُبلَّغ عنه"
Suggested fix:
Remove reportedPersonName from all individuals subtypes' detailFields and rely on entityName. Keep reportedPersonName only if it is repurposed as "nickname".

Medium Vehicle plate / app are collected twice

Files: EntityTypeNameStep.tsx, ReportDetailsStep.tsx, category-config.ts

For tourism > taxi and tourism > private_car, step 2 asks for "plate / app name" while step 4 asks for the same identifiers separately.

Step 2 label:
etnNameVehicle: "رقم اللوحة / اسم التطبيق"
Step 4 fields:
carPlate, appName, taxiNumber, carType, driverName, driverPhone
Suggested fix:
Change the vehicle entity-name label to "Office / company name" and keep the granular vehicle identifiers in report-details.

Medium reportedPersonNickname is declared but never used

Files: category-config.ts, ReportDetailsStep.tsx, ReviewStep.tsx, messages/*.json

The field exists in the type union, has Arabic/English labels, and is listed in the review mapping — but no subtype surfaces it and no input collects it. The review helper even maps its metadata key to undefined.

ReviewStep mapping:
reportedPersonNickname: undefined
Suggested fix:
Add a nickname input in ReportDetailsStep for relevant subtypes, or remove the dead field and its i18n keys.

Medium allegationPeriod is no longer collected

Files: validation.ts, state.ts, ExperienceStep.tsx

The schema and initial state still include allegationPeriod, but the active ExperienceStep does not ask for it.

Suggested fix:
Add a period input to ExperienceStep or remove the field from the schema/state if intentionally dropped.

Medium Review shows the country twice

Files: ReviewStep.tsx, LocationInfoStep.tsx

The review renders a standalone locCountry row and then a location row that already starts with the same country.

Current review output:
Country: Syria
Location: Syria — Homs
Nearest point: —
Suggested fix:
Remove the standalone locCountry row or merge country/city/nearest into one readable location line.

Medium Review does not show the media link

Files: ReviewStep.tsx, MediaEvidenceStep.tsx

MediaEvidenceStep collects reportMetadata.mediaLink, but the review Evidence group only renders sourceLinks and uploaded files.

Suggested fix:
Add a review row for reportMetadata.mediaLink so users can verify the public media link they entered.

Medium Location data is hardcoded in Arabic

Files: LocationInfoStep.tsx

The country and governorate dropdowns contain Arabic literals. In the English locale the dropdown still shows Arabic text.

Example:
{ value: "syria", label: "سوريا" }
Suggested fix:
Move country/governorate strings into messages/ar.json and messages/en.json and select by locale.

Medium English labels missing for flags and options

Files: category-config.ts, ExperienceStep.tsx, ReviewStep.tsx

COMMON_FLAGS, SUPPORTING_DOCUMENT_OPTIONS, and many subtype labels only define labelAr. The English UI falls back to Arabic text.

Example:
{ value: "owner", labelAr: "المالك / الشركاء" } // no labelEn
Suggested fix:
Add labelEn to all CategoryOption objects.
