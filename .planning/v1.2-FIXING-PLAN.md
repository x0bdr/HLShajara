# HLShajara v1.2 — Frontend Polish & Consistency

**Milestone:** v1.2 Frontend Polish  
**Goal:** Fix all UI/UX gaps identified in the 6-pillar audit, unify the design language across all pages, and make the platform feel like a single coherent product.  
**Baseline:** UI-REVIEW.md (2026-05-31) — Score 16/24  
**Target:** Score 22/24  
**Estimated effort:** 3–4 days  
**Priority:** High — current staging site looks inconsistent and unprofessional

---

## v1.2 Requirements

### UI-01 — Shared Page Shell (Header + Footer on ALL pages)
**Scope:** Every user-facing page must include the sticky Header and dark-green Footer.
**Pages affected:** `/record`, `/submit`, `/entity/[id]`, `/login`, `/dashboard`, `/reviewer`, `/mission`, `/faq`, `/policy`, `/terms`, `/privacy`, `/reply`
**Acceptance:**
- [ ] Create `PageShell` layout wrapper component
- [ ] Wrap all page components in `PageShell`
- [ ] Remove duplicate `main` wrapper inline styles from client pages
- [ ] Consistent `max-width: 1080px` across all pages
- [ ] Consistent padding (`20px` mobile, `32px` desktop)

### UI-02 — Dashboard Redesign (Token-Compliant)
**Scope:** Replace the current dashboard's bright traffic-light colors with the design token palette.
**Acceptance:**
- [ ] Remove all hardcoded hex colors (`#16a34a`, `#dc2626`, `#2563eb`, `#ca8a04`, `#7c3aed`)
- [ ] Use semantic tokens: `--green-700` (published), `--brick-500` (rejected), `--brass-600` (corrected), `--stone-600` (pending), `--brass-500` (under review)
- [ ] Replace `borderTop` colored bars with `.card` class + status dots
- [ ] Use `.ds-h1`, `.ds-lead`, `.ds-body` classes instead of inline styles
- [ ] Add loading skeleton using `.card` shape
- [ ] Add error state with retry button

### UI-03 — Evidence Card Click-Through
**Scope:** Wire up `EvidenceCard` `onOpen` handler to navigate to entity detail.
**Acceptance:**
- [ ] `ArchiveHome` passes `onOpen` to `EvidenceCard`
- [ ] Click routes to `/${locale}/entity/${e.id}`
- [ ] Same behavior on `/record` page
- [ ] Keyboard accessible (Enter/Space on focused card)

### UI-04 — Pagination for Archive Views
**Scope:** Add pagination to both homepage and `/record` page.
**Acceptance:**
- [ ] Backend: `GET /api/entity` supports `page` and `limit` query params
- [ ] Homepage: pagination controls below grid (Prev/Next or numbered)
- [ ] `/record` page: pagination controls below results
- [ ] Default limit: 12 (fits 2×6 grid)
- [ ] URL-synced page number for shareability

### UI-05 — Loading Skeletons & Error States
**Scope:** Replace plain-text "Loading..." with skeleton placeholders and proper error UI.
**Acceptance:**
- [ ] `RecordClient`: skeleton cards (6 placeholders) while fetching
- [ ] `EntityDetailClient`: skeleton panel while fetching
- [ ] `Dashboard`: skeleton stat cards while fetching
- [ ] All three: error state with message + "Retry" button
- [ ] Skeletons use `.card` shape with animated pulse (`opacity` animation)

### UI-06 — Unified Search/Filter Architecture
**Scope:** Pick ONE filter pattern and apply consistently.
**Decision:** Keep client-side filtering on homepage (for <100 records), server-side debounced on `/record` (for scalability).
**Acceptance:**
- [ ] Homepage filters remain client-side
- [ ] `/record` filters remain server-side debounced
- [ ] Both use identical filter UI (chips, search input, notice banner)
- [ ] Extract shared `FilterPanel` component from `ArchiveHome`
- [ ] `/record` page imports `FilterPanel` instead of native select elements

### UI-07 — Mobile Filter UX
**Scope:** Fix mobile filter sidebar stacking below results.
**Acceptance:**
- [ ] On screens <860px: filter sidebar collapses into a toggle button
- [ ] Toggle button shows active filter count badge
- [ ] Expanded filters show in a panel above results
- [ ] Smooth expand/collapse animation (200ms)

### UI-08 — Translation Gaps
**Scope:** Fix all hardcoded strings and missing translation keys.
**Acceptance:**
- [ ] Add `status_investigating`, `status_indicted`, `status_sanctioned`, `status_convicted`, `status_deceased` to messages
- [ ] Remove unused `status_confirmed`, `status_disputed` from messages
- [ ] Move filter-notice text to `messages/*.json`
- [ ] Move footer creed strings to `messages/*.json`
- [ ] Add `archiveTitle` key: "An archive of record" / "سجلٌّ موثَّق"
- [ ] Dashboard value labels use `useTranslations` instead of hardcoded text

### UI-09 — Typography Consistency
**Scope:** Remove inline styles, use `.ds-*` token classes everywhere.
**Acceptance:**
- [ ] `RecordClient`: replace inline styles with `.ds-*` classes
- [ ] `SubmitClient`: replace inline styles with `.ds-*` classes
- [ ] `EntityDetailClient`: replace inline styles with `.ds-*` classes
- [ ] `LoginClient`: replace inline styles with `.ds-*` classes
- [ ] `Dashboard`: replace inline styles with `.ds-*` classes
- [ ] Hero title weight: `700` for Arabic, `900` for English (or `700` both)
- [ ] Fix `[dir=rtl] .filter-group-label` to unset `text-transform: uppercase`

### UI-10 — Empty State Design
**Scope:** Improve "No results" and "No entries yet" states.
**Acceptance:**
- [ ] Homepage empty state: styled message + "Submit a report" CTA button
- [ ] `/record` empty state: styled message + search tips
- [ ] Empty state uses `.card` styling with centered content
- [ ] Illustration: seal icon + text (no decorative imagery per spec)

### UI-11 — Stats Bar Empty State
**Scope:** Handle zero-data gracefully.
**Acceptance:**
- [ ] When `entryCount === 0`, show `—` instead of `0`
- [ ] Or hide the stats bar entirely when no data exists
- [ ] Apply to both homepage and dashboard contexts

### UI-12 — Login UX Improvements
**Scope:** Basic accessibility and usability fixes.
**Acceptance:**
- [ ] Add "show password" toggle
- [ ] Add "remember me" checkbox (optional)
- [ ] Form field labels visible (not just placeholders)
- [ ] Error message styling matches design tokens (brick color, not default browser red)

## Out of Scope (v1.2)

- New features (search engine, AI classifier, partner API)
- Backend changes beyond pagination API params
- Mobile app or PWA
- Animation beyond subtle transitions
- Dark mode
- A/B testing or analytics

---

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Breaking existing pages during shell refactor | Medium | High | Test every page after adding Header/Footer |
| Dashboard redesign affects data readability | Low | Medium | Keep same stat layout, just change colors |
| Pagination changes break existing API consumers | Low | Medium | Default `page=1&limit=50` maintains backwards compat |
| Translation key changes break existing builds | Low | High | Run `next build` after every message change |
