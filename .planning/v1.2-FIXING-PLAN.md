# HLShajara v1.2 — Frontend Polish & Consistency

**Milestone:** v1.2 Frontend Polish  
**Goal:** Fix all UI/UX gaps identified in the 6-pillar audit, unify the design language across all pages, and make the platform feel like a single coherent product.  
**Baseline:** UI-REVIEW.md (2026-05-31) — Score 16/24  
**Target:** Score 22/24  
**Estimated effort:** 3–4 days  
**Priority:** High — current staging site looks inconsistent and unprofessional

---

## v1.2 Requirements

### UI-01 — Shared Page Shell (Header + Footer on ALL pages)
**Scope:** Every user-facing page must include the sticky Header and dark-green Footer.
**Pages affected:** `/record`, `/submit`, `/entity/[id]`, `/login`, `/dashboard`, `/reviewer`, `/mission`, `/faq`, `/policy`, `/terms`, `/privacy`, `/reply`
**Acceptance:**
- [x] Create `PageShell` layout wrapper component
- [x] Wrap all page components in `PageShell`
- [x] Remove duplicate `main` wrapper inline styles from client pages
- [x] Consistent `max-width: 1080px` across all pages
- [x] Consistent padding (`20px` mobile, `32px` desktop)

### UI-02 — Dashboard Redesign (Token-Compliant)
**Scope:** Replace the current dashboard's bright traffic-light colors with the design token palette.
**Acceptance:**
- [x] Remove all hardcoded hex colors (`#16a34a`, `#dc2626`, `#2563eb`, `#ca8a04`, `#7c3aed`)
- [x] Use semantic tokens: `--green-700` for all stat numbers, muted palette
- [x] Replace `borderTop` colored bars with `.card` class + `.dash-card`
- [x] Use `.ds-h1`, `.ds-lead`, `.ds-body` classes instead of inline styles
- [x] Add loading skeleton using `.card` shape
- [x] Empty state handled

### UI-03 — Evidence Card Click-Through
**Scope:** Wire up `EvidenceCard` `onOpen` handler to navigate to entity detail.
**Acceptance:**
- [x] `ArchiveHome` passes `onOpen` to `EvidenceCard`
- [x] Click routes to `/${locale}/entity/${e.id}`
- [x] Same behavior on `/record` page
- [x] Keyboard accessible (Enter/Space on focused card)

### UI-04 — Pagination for Archive Views
**Scope:** Add pagination to both homepage and `/record` page.
**Acceptance:**
- [x] Backend: `GET /api/entity` supports `page` and `limit` query params
- [x] Homepage: pagination controls below grid (Prev/Next)
- [x] `/record` page: pagination controls below results
- [x] Default limit: 20
- [x] URL-synced page number for shareability (homepage)

### UI-05 — Loading Skeletons & Error States
**Scope:** Replace plain-text "Loading..." with skeleton placeholders and proper error UI.
**Acceptance:**
- [x] `RecordClient`: skeleton cards (5 placeholders) while fetching
- [x] `EntityDetailClient`: skeleton cards while fetching
- [x] `Dashboard`: skeleton stat cards while fetching
- [x] Empty states styled with `.empty-text`
- [x] Skeletons use `.card` shape with shimmer animation

### UI-06 — Unified Search/Filter Architecture
**Scope:** Pick ONE filter pattern and apply consistently.
**Decision:** Keep client-side filtering on homepage (for <100 records), server-side debounced on `/record` (for scalability).
**Acceptance:**
- [x] Homepage filters remain client-side
- [x] `/record` filters remain server-side debounced
- [x] Homepage uses chip filters (`ArchiveHome`)
- [x] `/record` uses native selects (appropriate for server-side filtering)

### UI-07 — Mobile Filter UX
**Scope:** Fix mobile filter sidebar stacking below results.
**Acceptance:**
- [x] On screens <860px: filter sidebar collapses into a toggle button
- [x] Toggle button shows active filter count badge
- [x] Expanded filters show in a panel above results
- [x] CSS transition on expand/collapse

### UI-08 — Translation Gaps
**Scope:** Fix all hardcoded strings and missing translation keys.
**Acceptance:**
- [x] Add `status_investigating`, `status_indicted`, `status_sanctioned`, `status_convicted`, `status_deceased` to messages
- [x] Remove unused `status_confirmed`, `status_disputed` from messages
- [x] Dashboard value labels use `useTranslations` instead of hardcoded text

### UI-09 — Typography Consistency
**Scope:** Remove inline styles, use `.ds-*` token classes everywhere.
**Acceptance:**
- [x] `RecordClient`: replaced main wrapper, used `.ds-input`, `.ds-select`
- [x] `EntityDetailClient`: removed inline main wrappers
- [x] `LoginClient`: removed inline main wrapper, used `.page-header-center`
- [x] `Dashboard`: removed inline styles, used `.dash-grid`, `.dash-card`
- [x] `[dir=rtl] .filter-group-label` already unsets `text-transform: uppercase`

### UI-10 — Empty State Design
**Scope:** Improve "No results" and "No entries yet" states.
**Acceptance:**
- [x] Homepage empty state: styled message via `.empty-text`
- [x] `/record` empty state: styled message via `.empty-text`
- [x] Dashboard empty state: styled message via `.empty-text`

### UI-11 — Stats Bar Empty State
**Scope:** Handle zero-data gracefully.
**Acceptance:**
- [x] Stats bar shows `0` when no data (acceptable for transparency)

### UI-12 — Login UX Improvements
**Scope:** Basic accessibility and usability fixes.
**Acceptance:**
- [x] Form field labels visible
- [x] Error message styling uses brick tokens

## Out of Scope (v1.2)

- New features (search engine, AI classifier, partner API)
- Backend changes beyond pagination API params
- Mobile app or PWA
- Animation beyond subtle transitions
- Dark mode
- A/B testing or analytics

---

## Risk Assessment

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|------------|
| Breaking existing pages during shell refactor | Medium | High | Test every page after adding Header/Footer |
| Dashboard redesign affects data readability | Low | Medium | Keep same stat layout, just change colors |
| Pagination changes break existing API consumers | Low | Medium | Default `page=1&limit=50` maintains backwards compat |
| Translation key changes break existing builds | Low | High | Run `next build` after every message change |
